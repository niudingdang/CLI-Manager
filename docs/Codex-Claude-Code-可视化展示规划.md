# Codex 和 Claude Code 可视化展示规划

## 结论

下一步最值得做的不是继续增加零散的用量图，而是把一次 Codex / Claude Code 会话可视化成：

- 执行链
- 上下文链
- 代码变更链

这样 CLI-Manager 可以从“历史查看器”升级为“AI 开发过程审计台”。

## 当前基础

项目目前已经具备这些能力：

- 子任务可以打开新窗口查看。
- 可以读取历史记录。
- 可以查看用量统计。
- 可以查看上下文信息。
- 已经具备历史会话、Diff、统计看板等基础模块。

这些能力已经足够支撑下一阶段的可视化增强，不需要推倒重来。

## 功能方向优先级

| 优先级 | 可视化能力 | 展示内容 | 价值 | 复杂度 |
|---|---|---|---|---|
| P0 | 会话执行时间线 | 用户输入、模型回复、工具调用、文件读取、编辑、测试、报错 | 最直观地复盘 AI 做了什么 | 中 |
| P0 | 子任务树 / Agent 调用树 | 主任务、子任务、子任务结果、独立窗口入口 | 复用现有子任务窗口能力，形成总览 | 低-中 |
| P0 | 文件变更时间轴 | 每一轮改了哪些文件、每次 diff、测试是否通过 | 快速判断 AI 有没有乱改 | 中 |
| P1 | 上下文窗口仪表盘 | 上下文占用、压缩点、历史消息保留情况、Token 增长趋势 | 解决 Codex / Claude Code 上下文失控问题 | 中 |
| P1 | 工具调用瀑布图 | shell、read、search、patch、test 的耗时、成功/失败、输出摘要 | 定位卡顿、失败和低效步骤 | 中 |
| P1 | 成本 / Token 异常检测 | 按项目、模型、会话、日期统计异常消耗 | 在现有用量统计基础上增强 | 低 |
| P1 | 代码影响热力图 | 哪些文件最常被 AI 修改、哪些文件经常导致失败 | 适合长期项目治理 | 中 |
| P2 | Prompt 效果对比 | 同类任务下不同 prompt 的耗时、Token、成功率 | 帮助优化提示词 | 中-高 |
| P2 | 模型表现对比 | Codex vs Claude Code，不同模型在项目里的成功率、返工率 | 用于模型选择和成本控制 | 高 |
| P2 | Git 关联视图 | 会话、diff、commit、分支、测试结果的关联 | 支撑完整 AI IDE 管理闭环 | 中-高 |

## 推荐实施顺序

### 1. 会话执行时间线

核心目标：把一次 AI 开发过程拆成可点击、可追溯的事件流。

建议展示：

- 用户消息
- 模型回复
- 工具调用
- 文件读取
- 文件编辑
- 测试命令
- 报错信息

交互方式：

- 左侧显示时间轴。
- 点击节点后，右侧展示原始消息、命令输出、diff 或上下文片段。
- 支持筛选：只看错误、只看文件改动、只看工具调用。

### 2. 子任务树增强

当前已经支持子任务新窗口查看，下一步应增加总览树。

建议展示：

- 主任务和子任务层级。
- 每个子任务的状态。
- 子任务耗时。
- Token 消耗。
- 修改文件数量。
- 是否出现错误。
- 最后结论。

交互方式：

- 点击子任务节点，复用现有新窗口查看能力。
- 在主视图中显示整体任务拆分结构。

### 3. 上下文窗口视图

核心目标：让用户知道 AI 当前还“记得什么”。

建议展示：

- 每轮输入 Token。
- 每轮输出 Token。
- 累计 Token。
- 上下文压缩点。
- 摘要生成位置。
- 历史截断位置。
- 上下文增长趋势。

这对 Codex 和 Claude Code 都有实际价值，因为上下文失控是长会话里的核心问题。

### 4. 文件变更链

核心目标：把模型消息和代码 diff 建立双向关联。

建议能力：

- 从某条模型消息跳转到它造成的 diff。
- 从某个 diff 跳回触发它的消息。
- 从某个文件查看它在本次会话中被哪几轮 AI 修改过。
- 按时间展示每次 patch / edit。

这个能力比单纯展示最终 diff 更有价值，因为它可以解释“为什么改成这样”。

### 5. 工具调用诊断面板

核心目标：诊断 AI 执行过程中的失败、卡顿和低效。

建议展示：

- 每次 shell / read / search / patch / test 调用。
- 成功或失败状态。
- 执行耗时。
- 输出摘要。
- 错误输出。
- 慢命令高亮。
- 失败命令集中展示。

## 最小可行版本

建议 MVP 控制在以下范围内：

| 模块 | 必须能力 |
|---|---|
| 执行时间线 | 消息、工具调用、文件变更、错误 |
| 子任务树 | 主任务 / 子任务层级、状态、Token、耗时 |
| 上下文视图 | 每轮 Token、累计 Token、压缩点 |
| Diff 关联 | 消息和 diff 双向跳转 |
| 筛选能力 | 只看错误、只看文件改动、只看工具调用、只看子任务 |

## 不建议优先做

| 不建议优先做 | 原因 |
|---|---|
| 很花的知识图谱 / 网络图 | 好看但信息密度低，容易变成摆设 |
| 复杂自动评分系统 | 成功率和质量分很难客观定义，早做容易误导 |
| 大而全的 AI 看板首页 | 当前核心价值应放在单次会话复盘 |
| 过早做跨模型排行榜 | 数据口径不稳定，Codex 和 Claude Code 行为差异大 |

## 产品定位

这条路线的核心不是“统计更多数据”，而是让用户能回答这些问题：

- AI 到底做了什么？
- 它为什么改这个文件？
- 哪一步开始出错？
- 哪个子任务最耗 Token？
- 当前上下文是否已经失控？
- 这次会话产生的代码改动是否可追溯？

## 一句话方向

把 Codex / Claude Code 的黑盒过程拆成可点击、可追溯、可审计的开发事件流。
